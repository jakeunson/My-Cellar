# 🤖 AI Coding Agent Behavior Rules (에이전트 행동 규칙)

이 파일은 **My Cellar (Jetpack Compose 주류 테이스팅 저널)** 프로젝트를 위한 AI 코딩 에이전트 전용 지침 및 행동 규칙 문서입니다. 에이전트는 향후 이 프로젝트의 코드를 수정하거나 기능을 추가할 때 아래의 규칙을 엄격히 준수해야 합니다.

---

## 1. 프로젝트 핵심 목적 및 정체성 (Identity)
* **목적**: 이 앱은 안드로이드 **Material Design 3 (M3)** 및 **Jetpack Compose**를 기반으로 작동하는 주류 테이스팅 저널 앱의 구조 설계, 화면 흐름 시뮬레이션 및 실제 빌드 가능한 코틀린(Kotlin) 코드를 제공하는 통합 도구입니다.
* **정체성**: 
  * 왼쪽에는 실제 작동을 시각화하는 모바일 형태의 **Phone Simulator (React)**가 탑재되어 있고, 오른쪽에는 이에 동기화되는 **Code Viewer (Kotlin)**가 배치되어 있습니다.
  * 시뮬레이터의 화면 흐름 및 상태 변화는 우측 코드 뷰어에 표시되는 파일과 밀접하게 연동(Sync)되어야 합니다.

---

## 2. 디자인 및 레이아웃 원칙 (Design Principles)
* **비주얼 테마**: 
  * 차분한 **Slate(회색/남색 계열)** 캔버스와 꿀빛 위스키 골드 및 참나무 갈색인 **Amber/Oak** 계열 악센트 컬러가 조화롭게 녹아든 고대비 라이트/네추럴 스타일을 유지합니다.
  * 견고한 기하학적 형태의 굵은 보더라인(Slate-900 위주의 테두리, `border-4` 등)과 개성 있는 미니멀/네오브루탈리즘 디자인 아이덴티티를 보존합니다.
* **타이포그래피**:
  * 디스플레이 및 헤더 영역에는 기하학적이고 현대적인 폰트 굵기를 사용하고, 코드 영역은 일관된 모노스페이스(`font-mono`) 폰트를 사용합니다.
  * 한국어 텍스트와 가독성 있는 폰트 스케일링, 충분한 여백 및 대비(Contrast)를 확보하여 탁월한 시각적 완성도를 제공합니다.
* **아이콘**:
  * 모든 리액트 UI 아이콘은 `lucide-react`에서 직접 명명된 컴포넌트로 임포트하여 사용하며, 임의의 인라인 SVG 사용은 금지합니다.

---

## 3. Jetpack Compose & 안드로이드 코드 작성 규칙
우측 `CodeViewer`에 표시되는 Kotlin/Compose 코드는 안드로이드 스튜디오에 붙여넣었을 때 즉시 컴파일 가능해야 합니다.
* **Material Design 3 (M3) 준수**: `androidx.compose.material3` 패키지의 최신 컴포넌트(`Scaffold`, `MediumTopAppBar`, `Button`, `FilterChip`, `Slider`, `Card`)를 충실히 따릅니다.
* **상태 관리 및 단방향 데이터 흐름 (UDF)**:
  * 컴포저블 내에서 직접 상태를 조작하지 않고, 상태 호이스팅(State Hoisting) 기법을 사용하여 상태(`remember { mutableStateOf(...) }`)와 이벤트 콜백(`onValueChange`, `onClick`)을 깔끔히 분리합니다.
* **네비게이션 구조 설계**:
  * 단일 액티비티 내에서 선언형 네비게이션(`NavHost`, `composable`, `NavController`) 구조를 가지며, 라우트 네이밍(`"home"`, `"add"`, `"detail/{id}"`) 규칙을 명확히 정의합니다.

---

## 4. 기능 추가 및 수정 시 에이전트 금지 사항 (Strict Negative Constraints)
* **임의적인 테마 파괴 금지**: 기껏 구성해 둔 세련된 고대비 Slate & Amber 조합의 톤앤매너를 깨트리거나 흔한 보라색/그라데이션 템플릿 형태로 회귀하지 마십시오.
* **시스템 로깅/기계 데이터 추가 자제**: UI 여백이나 가두리에 불필요한 시스템 메타데이터, 로그 출력, 시뮬레이션 핑 상태(예: `● SYSTEM LIVE`, `PORT: 3000`)를 지저분하게 추가해 주는 "기술 시늉(Tech-Larping)"은 철저히 배제하고, 깔끔한 사용자 환경을 완성하십시오.
* **사용자 의도 초과 구현 금지**: 사용자가 요청한 스코프 안에서만 정교하고 기품 있는 수정을 가하고, 요구하지 않은 다크모드 토글 스위치나 다른 보조 차트 등의 기능을 마음대로 추가하여 볼륨을 늘리지 마십시오. 장인 정신은 기능의 양이 아닌 마감의 퀄리티에서 나옵니다.

---

## 5. 변경 및 검증 프로세스
* 새로운 기능이나 시뮬레이션 상태가 추가되면 반드시 React 측 컴포넌트(`App.tsx`, `PhoneSimulator.tsx`, `CodeViewer.tsx`)와 우측 Kotlin 가상 코드 파일 내의 내용을 함께 최신화하여 모순이 없도록 해야 합니다.
* 코드 수정 작업이 끝날 때마다 반드시 `lint_applet`과 `compile_applet`을 수행하여 빌드 에러 및 의존성 문제가 없는지 검증하십시오.
